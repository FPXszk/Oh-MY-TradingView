# 最新ワークフロー失敗調査計画

## 目的

GitHub Actions の最新失敗 workflow run を特定し、失敗ジョブ・失敗ステップ・根因候補を整理する。必要であればローカル再現可能性まで確認するが、この計画では原則として「調査」と「原因整理」までを対象とし、修正実装は含めない。

- 既存 completed plan に workflow / CI 調査系の履歴はあるが、今回の対象は「現時点での最新失敗 run」であり、過去 run の修正作業とは切り分けて扱う

## 確認対象ファイル・対象物

- 確認: GitHub Actions の最新失敗 workflow run
- 確認: `.github/workflows/*.yml` のうち、最新失敗 run に対応する workflow 定義
- 確認: 失敗ステップで参照されている repo 内ファイル
- 参照のみ: `docs/exec-plans/completed/` 配下の関連する過去計画（必要な場合のみ）

## スコープ

### 実施すること

- 最新の失敗 run を特定する
- run / job / step の失敗内容を高シグナル順に確認する
- root cause の仮説を整理する
- 必要なら最小限のローカル再現コマンドを確認する
- ユーザーに調査結果、影響範囲、次の修正候補を報告する

### 実施しないこと

- ユーザー承認なしで本番コード・workflow 定義・設定ファイルを変更しない
- 原因未確定のまま広範囲な修正やリファクタを始めない
- unrelated な失敗や古い run の掘り下げを並行して広げない

## 実装内容と影響範囲

- GitHub Actions の最新失敗 run のみを対象に調査する
- 影響範囲の評価対象:
  - 対象 workflow 名
  - 失敗 job / step
  - 関連する `.github/workflows/*.yml`
  - 失敗ステップが実行するスクリプト・設定・テスト

## 調査方針

1. まず最新失敗 run の一覧と run summary を確認し、対象 workflow・branch・SHA・event を確定する
2. 次に失敗 job / step の要約を確認し、必要時のみ詳細ログへ降りる
3. 失敗が設定・権限・環境・コード・外部依存のどれに属するかを切り分ける
4. ログだけで根因が十分明確でない場合のみ、workflow と同等のローカル確認に進む
5. 調査結果を「事実」「推定」「不足情報」に分けて報告する

## TDD / 検証戦略

この依頼は現時点では調査タスクであり、修正実装を前提としない。

ただし、失敗原因の確認にローカル再現が必要な場合は以下の順序を使う:

### RED

- GitHub Actions の最新失敗 run を既存の失敗事実として扱う
- 必要に応じて workflow の失敗コマンドをローカルで再実行し、同種の失敗を確認する

### GREEN

- 本計画の範囲外。修正実装が必要と判断した場合は、別途修正計画または追記承認を取って進める

### REFACTOR

- 本計画の範囲外

### カバレッジ方針

- 調査段階では coverage 向上を目的にしない
- 修正が必要になった場合に、対象に応じて unit / integration / E2E の追加要否を整理する

## 想定コマンド

```bash
gh run list --limit 10
```

```bash
gh run view <run-id>
```

```bash
gh run view <run-id> --log-failed
```

必要に応じて:

```bash
sed -n '1,220p' .github/workflows/<workflow>.yml
```

```bash
<workflow 内の失敗ステップと同等のローカルコマンド>
```

## リスク / 注意点

- 最新失敗 run が scheduled workflow の場合、外部サービス状態や secrets 期限が根因で、ローカル再現できない可能性がある
- 同時刻帯に複数 workflow が失敗している場合、「最新」の定義を run 作成時刻で固定して扱う
- `gh` で取得できる要約だけでは不十分なとき、ログ量が多く調査が広がりやすいので、対象 job を絞って掘る
- 調査だけで修正方針が一意に定まらない場合、その時点で止めて選択肢を報告する

## 実装ステップ

- [ ] `gh run list --limit 10` で最新失敗 run を特定する
- [ ] 対象 run の workflow 名、branch、event、SHA、失敗時刻を記録する
- [ ] `gh run view <run-id>` と必要に応じて `--log-failed` で失敗 job / step を特定する
- [ ] 対象 workflow 定義と関連ファイルを確認し、失敗の直接原因候補を整理する
- [ ] ログのみで不十分な場合は最小限のローカル再現可否を確認する
- [ ] 調査結果を「原因候補」「影響範囲」「追加で必要な修正有無」に分けて報告する

## 完了条件

- 最新失敗 run が一意に特定されている
- 失敗 job / step と根因候補が説明できる
- 影響範囲と次のアクション候補が明確になっている
- 調査結果が、修正実装に進むかどうか判断できる粒度で整理されている
