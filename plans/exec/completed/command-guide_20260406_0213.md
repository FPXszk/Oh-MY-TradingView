# Exec Plan: TradingView cleanup / worker2 manual launch command guide

## Problem

ユーザー要望は、TradingView の運用で必要なコマンドを 1 つの `docs/command.md` に整理することです。対象は以下の 3 系統です。

- 既存の TradingView プロセスを**安全に全停止**する手順
- worker2 を**手動で起動**する手順
- WSL / CLI から**疎通確認と状態確認**を行う手順

加えて、実装後は変更をコミットし、リモートへ push できる状態まで持っていく必要があります。ただし、repo のワークフロー上、まず exec-plan を作成し、ユーザー承認後に実装へ進む必要があります。

## Approach

既存の worker2 handoff / session log / ドキュメントを再確認し、すでに安定動作が確認されているコマンドだけを `docs/command.md` に整理します。  
ドキュメントは「停止 → 起動 → 確認」の順で並べ、**PID を明示して停止する安全な手順**を採用します。  
実装はドキュメント追加を中心にし、必要最小限で README や関連 doc への導線追加が必要かも確認します。

## Test strategy

ドキュメント中心の変更だが、RED -> GREEN -> REFACTOR の形で進めます。

### RED

- 既存 docs / logs から必要コマンドが散在している状態を確認
- 停止 / 起動 / 確認コマンドが単一ドキュメントにまとまっていないことを確認

### GREEN

- `docs/command.md` に以下が揃っていることを確認
  - 全停止のためのプロセス列挙コマンド
  - PID 指定停止コマンド
  - worker2 手動起動コマンド
  - portproxy 確認コマンド
  - WSL からの `curl` / `node src/cli/index.js status` 検証コマンド

### REFACTOR

- コマンド順序を「停止 → 起動 → 検証」に整理
- 危険な停止方法（`pkill` / `killall` 相当）を避ける注意書きを最小限で明記
- 重複する説明を削り、再利用しやすい形へ整える

## Validation commands

### Documentation / repository checks

```bash
git --no-pager diff -- docs/command.md README.md docs/DOCUMENTATION_SYSTEM.md
```

### Worker process discovery / cleanup (PowerShell)

```powershell
Get-CimInstance Win32_Process | Where-Object { $_.Name -eq "TradingView.exe" }
```

```powershell
Get-CimInstance Win32_Process | Where-Object { $_.Name -eq "TradingView.exe" -and $_.CommandLine -match "9224" }
```

```powershell
Stop-Process -Id <PID>
```

### Worker2 manual launch (PowerShell)

```powershell
Start-Process -FilePath "C:\TradingView\TradingView.exe" -ArgumentList "--remote-debugging-port=9224","--user-data-dir=C:\TradingView\profiles\worker2","--in-process-gpu"
```

```powershell
& "C:\TradingView\TradingView.exe" --remote-debugging-port=9224 --user-data-dir=C:\TradingView\profiles\worker2 --in-process-gpu
```

### Portproxy / connectivity verification

## Risks

- session log と実環境でコマンド差分がある可能性
- `docs/command.md` の配置を root と想定している
- PID 停止手順を誤って一般化すると危険
- WSL 側 IP / portproxy の状態が環境依存で変わる可能性
- ドキュメント更新のみでも、関連導線の不足で discoverability が低い可能性
